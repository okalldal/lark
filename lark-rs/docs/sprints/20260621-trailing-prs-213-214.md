## Run shape

This was a **second-half-only** invocation: review + stage + omnibus for two already-open,
already-CI-green trailing `/next-task` PRs (#213→#176, #214→#210), not a full survey/dispatch
sprint. Both were dispatched and built by earlier `/next-task` sessions before this run.

## RETRO — orchestrator

- **Adopting pre-existing `master`-targeting PRs as sprint children.** #213/#214 were born as
  normal `/next-task` PRs: base `master`, bodies carrying `Closes #176`/`Closes #210`. The
  sprint kit assumes children are *created* against the sprint branch with `Refs`/`Part of` and
  no closing keyword. Adopting them required retargeting their base to the integration branch
  (which neutralizes the closing keyword on a non-default base) and letting the omnibus own the
  real `Closes`. Suggest a short §-note in `start-sprint.md` for "adopting trailing
  master-targeting PRs into a sprint's second half."
- **Stacked child PRs + merge-method interaction.** #214 was a strict commit-stack on top of
  #213 (shared the exact commit `2a97343`). Staging order was therefore forced (#213 first).
  Critically, **squash-merging the lower PR (#213) would have discarded the shared commit and
  reintroduced a phantom CLAUDE.md conflict** on #214; using a real **merge commit** preserved
  `2a97343` as a common ancestor and kept #214 conflict-free. Suggest the staging-queue note
  warn: for stacked children, stage in stack order and prefer a merge commit over squash so the
  shared base commit is preserved.
- **Branch-push restriction made child-branch conflict resolution impossible for the
  orchestrator.** This session was authorized to push only to the integration branch, not the
  child PR head branches — so the §6 "dispatch a worker to resolve the conflict in-worktree and
  push to the child branch" escape hatch was unavailable. Mitigated by **simulating the merges
  locally on a throwaway branch first** to prove the chosen staging order was conflict-free
  before any server-side merge. Suggest the kit note this "simulate-before-server-merge" safety
  check as the right move whenever child-branch pushes are restricted.

## RETRO — review sub-agent (#214/#210)

- The stacked-PR relationship was invisible from #214's metadata alone (same base SHA as #213,
  separate branch); it only surfaced by diffing the two PRs' file lists against `master`. A
  sprint-child review packet for a stacked PR should state the parent-PR dependency explicitly,
  not just "sibling fix in the same file."
- #214's PR body described only the #210 dedup, giving no hint the diff also carried the full
  #176 changeset — a reviewer trusting the body would under-estimate the diff's true scope.
  Worth a kit note: stacked PRs should declare their parent PR in the body.
