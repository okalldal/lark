# ADR-0020: LALR error recovery over a postlex (Indenter) hook â€” mirror Python's per-resume `Indenter.process` reset

- **Status:** Proposed (2026-06-19; pending architect ratification)
- **Date:** 2026-06-19

## Context

Error recovery (#43/#93/#167) shipped **LALR-without-postlex only**:
`ParsingFrontend::parse_recovering` returned `GrammarError::Other` whenever a
postlex hook was present, with the message *"error recovery requires
parser='lalr' without a postlex hook"*. The stated reason (mod.rs doc-comments)
was that single-token-deletion recovery deletes tokens from the stream, which
"could desync the indenter's synthetic INDENT/DEDENT injection."

Issue #94 scoped (architect verdict, 2026-06-18) to **sub-target 1 only**:
lift that restriction for the LALR + Indenter postlex path. Earley/CYK recovery
were dropped as won't-do â€” Python exposes no `on_error` there, so they have no
oracle and are unfalsifiable.

The desync worry turned out to be the wrong mental model. Tracing Python Lark
(the oracle) settled three load-bearing facts:

1. **The indenter is upstream of deletion.** Python wires
   `lexer â†’ PostLexConnector(postlex) â†’ parser`, and `on_error`/`resume_parse`
   operate on the *post-indenter* token stream. INDENT/DEDENT are injected over
   the clean lex; the parser deletes from the already-indented stream. A deleted
   token never reaches the indenter, so the feared desync does not occur.
2. **But the indenter *resets* on every resume.** `Indenter.process` sets
   `indent_level = [0]` and `paren_level = 0` at the **start of every call**, and
   `resume_parse` re-invokes `process`. So after the parser deletes an offending
   token, a *fresh* indenter continues pulling real tokens from the lexer's
   current position with a cleared indent/paren stack. This is observable: a
   multi-deletion recovery never re-emits the trailing DEDENTs that close a block,
   so Python re-raises at `$END` where a naive "indent the whole stream once"
   model would (wrongly) recover a tree.
3. **Char skips reset the indenter too.** Python routes *both* `UnexpectedToken`
   (token deletion) and `UnexpectedCharacters` (char skip) through `resume_parse`,
   so a char skip inside an indented block also resets the indenter.

## Decision

Implement LALR + postlex recovery as a **faithful mirror of `resume_parse`**,
not as an "indent once, then recover" approximation:

- Drive recovery through the existing streaming indenter (`IndenterStream` via
  `PostlexContextual`), which sits upstream of the parser's token deletion exactly
  as Python's `PostLexConnector` sits upstream of `resume_parse`.
- On **every** recovery resume â€” a token deletion *and* a char skip â€” reset the
  indenter (`indent_stack = [0]`, `paren_level = 0`, drop the queued tokens),
  leaving the underlying lexer position untouched. This is the generic
  `TokenSource::on_delete` hook (no-op for non-postlex sources) plus the same
  reset in `PostlexContextual::skip_char`.
- Recover over the **contextual** stream when the contextual lexer is load-bearing
  (a `ContextualRecovering` inner source with root-lexer fallback, #166), and over
  the **basic/global** stream for the explicit `lexer='basic'` postlex driver (a
  new lazy `BasicRecovering` inner source). Both feed the *same* streaming
  indenter, so char skips interleave with the indenter reset identically.
- An indenter error (e.g. a `DedentError`: a dedent to an unknown column) is raised
  by the postlex hook itself, before any parser token error â€” surface it as a hard
  `LarkError`, mirroring Python re-raising it through the postlex generator without
  consulting `on_error`.
- Relax exactly one guard during recovery: an unbalanced closing bracket saturates
  `paren_level` at 0 instead of erroring (the `recovering` flag on `IndenterStream`).
  Python's `paren_level -= 1; assert >= 0` is *lazy* â€” it runs on the iteration
  after the close bracket is yielded â€” and during recovery the generator is
  abandoned (a fresh one replaces it) before that assert fires, so the underflow is
  never observed. A clean parse keeps the hard error.

The refusal now covers Earley/CYK only; the message becomes *"error recovery
requires parser='lalr'"*.

## Consequences

- **Reverses a documented restriction.** This ADR supersedes the prior "no
  recovery with a postlex hook" choice (it lived only in mod.rs doc-comments and
  the error string, never an ADR). Every LALR configuration now supports recovery.
- **Oracle-pinned, byte-for-byte.** `tests/fixtures/oracles/indenter_recovery/`
  (generated by `generate_indenter_recovery`, `on_error=lambda e: True`) freezes
  Python's recovered tree + deletion/skip count for the `indent`, `indent_paren`,
  and contextual-load-bearing `indent_context` grammars, replayed under both the
  contextual and basic postlex paths. Cases deliberately exercise multi-deletion,
  char-skips inside indented blocks, paren underflow, premature-`$END` re-raise,
  and the `DedentError` hard-error path â€” the spots where the naive model diverges.
- **Altitude: one streaming-indenter machine.** `PostlexContextual<S: TokenSource>`
  is generic over its inner source, so the clean parse (`Contextual`), contextual
  recovery (`ContextualRecovering`), and basic recovery (`BasicRecovering`) all
  share the same per-resume-reset logic â€” no special-cased "postlex recovery"
  branch threaded through the parser loop.
- **Public API.** `Lark::parse_with_recovery` / `parse_on_error` now succeed on
  LALR + Indenter grammars instead of erroring. No signature change; a previously
  failing configuration starts working. `escalate`-tier (the architect approves via
  the omnibus) since it widens recovery's supported surface.
- **Earley/CYK stay refused** â€” unfalsifiable without an `on_error` oracle, per the
  #94 verdict. The refusal message no longer mentions postlex.
