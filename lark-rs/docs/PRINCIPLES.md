# PRINCIPLES — the lark-rs constitution

> **Status: DRAFT proposal.** This is the grounding document for autonomous
> development of this repo. It is *yours* (the architect's) to own and edit; agents
> read it and cite it but never rewrite it (§7). Everything else in the autonomy
> kit — the `/roadmap`, `/triage`, `/review-pr`, `/next-task`, `/finish-task`
> commands, the label schema, the ADR log — exists to enforce or extend what is
> written here.

---

## 0. The thesis

The repo already proved one move: **oracle-first testing.** It worked because it
converted a question that *would* need human review — "is this code correct?" —
into a falsifiable gate an agent checks itself against ("does it match Python
Lark?"). The agent answers without the architect in the loop.

This document generalizes that move to every class of decision:

> **The boundary of safe autonomy is the boundary of what we have made
> falsifiable.** An agent may decide anything it can ground against an oracle, a
> gate, or a written rule. Everything else it must escalate — or the architect
> must build the missing gate.

The architect's job at this altitude is **building gates and handling
escalations**, not making individual decisions. This constitution is the gate for
the decisions that don't have a test-shaped one.

Corollary — **all durable state lives in git + GitHub** (this doc, the ADR log,
issues, labels, PRs). A session is a *stateless operator* over that state: it
reads, acts, writes back, and leaves nothing important in conversation memory.
That is what makes manual, vanilla-Claude-Code sessions sufficient.

## 2. Invariants — never violate

These are load-bearing. A change that breaks one is wrong even if CI is green;
if CI *lets* you break one, the gate is buggy — fix the gate, file an issue.

1. **Oracle-first.** No grammar feature without a failing oracle test *before*
   implementation. No bug fix without a reproduction that fails first. (Testing
   Philosophy, `lark-rs/CLAUDE.md`.)
2. **Python Lark is the oracle.** Expected trees/errors come from Python Lark via
   `tools/generate_oracles.py` — never hand-authored to match the Rust output.
3. **Never hand-edit generated artifacts.** Anything under
   `tests/fixtures/oracles/` or `tests/standalone/*.rs` is regenerated by the
   tools (`.claude/settings.json` denies direct edits). See `/regen-oracles`.
4. **Never regress a green corpus.** JSONTestSuite stays 293/293; the compliance
   banks only shrink their XFAIL lists, never grow them outside a recorded
   burndown (`/xfail-burndown`).
5. **Performance claims are deterministic.** Super-linearity is proven and gated
   on the `perf-counters` work counters, never wall-clock (`BENCH.md`). A
   suspected pathology becomes a committed scaling test *before* a fix, and the
   fix targets the cause the profiler names.
6. **Upstream grammars are verbatim.** Wild-bank grammars are vendored unedited;
   divergence is recorded in `meta.json`, never silently patched. An `alt_grammar`
   must be tree-identical to the original's Python oracle on every input.
7. **Falsifiable over aspirational.** Prefer a quality you can *check* over one you
   can only *assert*. This is the meta-invariant the other six are instances of.

## 3. Defaults — deviate only with an ADR

Sensible unless there's a recorded reason otherwise. To deviate, write an ADR
(`docs/decisions/`, §7) — that is the whole cost, and it is what lets the
architect audit the deviation later instead of catching it in review.

- **End-to-end over unit tests** — the oracle checks the full pipeline.
- **Small blast radius over big-bang rework** when value is comparable. A
  large-blast-radius change is allowed but must say so and lean on the compliance
  banks as the net (`/next-task` rubric).
- **Python `re` dialect fidelity over `regex`-crate convenience** where the two
  disagree (ADR-0004; notes in `CLAUDE.md`).
- **No runtime backtracking engine** — bounded lookaround is *lowered* into the
  DFA; out-of-scope shapes take a categorized refusal through the single seam
  (`docs/LOOKAROUND_SCOPE.md`).
- **A new parser×lexer configuration is a new driver impl, not new match arms.**

## 4. The decision taxonomy — how an agent decides whether it may decide

For any decision, find its row. This is the operational form of the thesis.

| The decision is grounded by… | Then the agent… | Examples |
|---|---|---|
| An **oracle / gate / bank** (correctness, perf envelope, regression, CI) | **Decides freely**, self-checks | feature impl, bug fix, perf fix behind a gate, refactor with banks green |
| A **written rule + a judgement call** (a §3 default, a triage rubric) | **Decides, records an ADR** if it deviates from the default | architecture choice, priority ordering, "is this PR mergeable?" |
| **Nothing falsifiable** — it's product direction or taste | **Escalates** (§5) | what to build next, a new phase, a genuine tradeoff with no oracle |

When unsure which row you are in, you are in the bottom row. Escalate.

### The decision lens — how to reason inside the middle row

The taxonomy *routes* a decision; it doesn't say how to weigh the "decide + ADR"
row. These are the dimensions a non-trivial design decision is judged on. **This is
not a scorecard — you do not tally it.** Each dimension is either a *gate* (then the
gate decides; don't argue it in prose — run or add the gate) or *judgment-only*
(then your reasoning is the artifact, and you name it in the ADR).

| Dimension | Its gate (if any) | When it's judgment-only |
|---|---|---|
| **Correctness & Python parity** | Oracle + compliance/wild banks (§2.1–2.2) — *the dominant axis* | A legitimate superset-of-Lark divergence → `needs-decision`, not a free win (#101, #159) |
| **Performance & complexity class** | Work-counters + scaling gates (§2.5) | A new algorithm with no envelope yet → propose the gate *first* |
| **Maintainability & simplicity** | `/code-review`; consolidate-seams-before-features (ADR-0015); new-driver-not-match-arms (§3) | Reuse vs. new abstraction is mostly taste — argue it in the ADR |
| **Portability (PyO3 / WASM / C / standalone)** | CI binding jobs; const-bakeability | "Does this bake into every target?" — e.g. the Tree worklist rewrite (#151) was a WASM-stack call |
| **Security / robustness on adversarial input** | Scaling gates catch *algorithmic-complexity* DoS | Resource bounds on untrusted input via the public API — **largely ungated; judgment + flag the gap** |
| **API & grammar-author ergonomics** | — none | Error-message quality, modifier semantics — **ungated; judgment-only** |

Two rules keep this from sliding into the aspirational hand-waving §2.7 forbids:

1. **A gated dimension is decided by its gate, not by argument.** If you're weighing
   it in prose, you're missing a gate — add it.
2. **A material judgment-only dimension must be *named* in the ADR.** That naming is
   the auditable, falsifiable substitute for a test.

On NFRs: this *is* the NFR layer, built the repo's way — each quality resolves to a
gate or a named ADR axis, not a prose aspiration (an external taxonomy like ISO/IEC
25010 is at most a one-time coverage check, never working vocabulary). The actionable
residue: the two ungated bottom rows — **security/resource bounds on untrusted input**
and **grammar-author ergonomics** — should become tracked issues and, eventually,
gates.

## 5. Escalation — cheap, rare, batched

The escalation channel must be cheap enough that erring toward it costs little, so
agents never *guess* on direction.

- **Asynchronous / batched:** open or label an issue `needs-decision`. This is the
  architect's inbox; `/triage` and `/roadmap` surface it. The canonical shape is a
  semantic or policy fork an implementation can't settle — see `LABELS.md` and the
  live `needs-decision` queue for current examples.
- **Synchronous (only when it blocks the current session):** `AskUserQuestion`
  with enough context to answer without scrolling back, and a recommended option
  first. Use this sparingly — most direction questions can wait in the
  `needs-decision` queue.
- **Never** resolve a `needs-decision` item by picking the option that is easiest
  to implement. An agent that finds *only* `needs-decision` work surfaces it; it
  does not invent groundable work to avoid asking.

## 6. Definition of Done & merge tiers

**Definition of Done** (the gate before an issue closes / a PR merges):

1. **Verified the right way for its type.** *Code/behavior change:* a failing
   oracle, repro, or scaling gate existed *first* and now passes (invariant 1).
   *Docs/governance/triage change:* the affected command or policy path was walked
   through by hand and any contradictions resolved — there is no failed-first test
   to expect, and demanding one is the wrong gate.
2. CI is green — the `pull_request` run, which is the full gate.
3. `/code-review` ran on the diff and findings are addressed.
4. Oracles / `STATUS.md` are fresh (the CI freshness gate is happy).
5. Out-of-scope discoveries are filed as issues (§7), not left in the diff.
6. An ADR exists if a §3 default was deviated from.
7. The originating issue's *done-when* is satisfied, and the PR says `Closes #N`.

**Merge tiers** (full rationale: ADR-0016, `docs/decisions/`). The line is exactly
*"is correctness fully captured by an existing gate?"*

> **Activation gate.** ADR-0016 is **Proposed**, not accepted. Until the architect
> accepts it, `auto` is a *recommendation only*: `/review-pr` posts the verdict but
> **the architect merges every PR**. The tiers below define what auto-merge *will*
> mean once enabled — after a few successful verdict-only dry runs.

- **`auto` (agent may merge, once enabled)** when DoD is met and the change is a
  bugfix-with-oracle, xfail burndown, perf fix behind a gate, a refactor with no
  public-API change and banks green, or **trivial docs** (typo, link, status
  refresh, non-normative clarification). Here CI *is* the oracle; a human merge adds
  latency, not safety.
- **`escalate` (architect merges)** for new public API, new grammar-feature
  semantics, architecture changes, **any governance/policy doc** (this constitution,
  ADRs, command behavior, `LABELS.md`, roadmap, `CLAUDE.md`, public claims, or
  architectural responsibility boundaries), or anything labeled `needs-decision`.
  `/review-pr` may *approve* these but never merges them.

## 7. The issue contract

A well-formed issue (what `/roadmap` drafts and `/finish-task` files) carries, in
the de-facto shape already used across this backlog:

- **Done-when** — the falsifiable acceptance check (ideally an oracle/test name).
- **Priority** — `prio:now | prio:next | prio:later` (§ LABELS.md).
- **Files** — the likely blast radius.
- **Notes / Decision needed** — context, and any unresolved fork that makes it a
  `needs-decision` item rather than `good-autonomous`.

Out-of-scope rule: when an agent finds a bug or follow-up outside the current
task, it **files an issue** (with the above), never silently fixes it (blast-radius
discipline) and never silently drops it. Much of the current follow-up backlog was
filed this way mid-task — that is how the backlog stays honest.

## 8. How this constitution evolves

This document is not frozen and it is not edited by agents. It evolves through one
loop:

```
agent makes a semi-groundable call  →  records an ADR (docs/decisions/)
        →  architect reads ADRs in arrears (batched, not per-PR)
        →  architect promotes any correction into §2/§3 here
        →  every future agent is now grounded by the sharpened rule
```

That loop is the answer to "did I formulate my vision well enough?" — you don't
have to get §2/§3 right up front; each correction tightens them. The ADR log is
the audit trail that lets you stay in control *without* being in the loop.

## 9. Operating safety net — rollback, audit, policy isolation

Three hard rules protect master and keep agents from quietly widening their own
authority:

- **Policy changes ride their own PR.** A code/behavior PR must **not** also edit
  this constitution, the merge tiers, command behavior, or label semantics. Agents
  do not change the rules they operate under while shipping work; governance changes
  are separate, `escalate`-tier PRs.
- **Rollback first, diagnose second.** An auto-merged PR that turns master red is
  reverted immediately (`git revert`) and an incident issue opened. *Then:* if the
  failure escaped a gate, add or fix the gate before retrying; if it was a mis-tier
  (an `escalate` change merged as `auto`), record an ADR or tighten §6.
- **Periodic autonomy audit.** Every ~10–20 merged PRs, the architect (or an audit
  pass) reviews the auto-merges, escalations, reverts, and "agent guessed wrong"
  cases. The ADR log captures *explicit* deviations; this audit catches the
  *routine* auto-tier mistakes the log would miss.
