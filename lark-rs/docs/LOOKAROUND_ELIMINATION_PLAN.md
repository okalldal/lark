# Lookaround elimination plan

*Status: implementation roadmap. Rationale and decision in
[`LOOKAROUND_STRATEGY_ANALYSIS.md`](LOOKAROUND_STRATEGY_ANALYSIS.md).*
*Supersedes the former `LEXER_DFA_PLAN.md` (the Pike-VM lowering strategy), removed
in this PR — it remains in git history and on the closed
[PR #110](https://github.com/okalldal/lark/pull/110) branch as the spec for the
shelved fallback engine (Option H).*
*Date: 2026-06-07.*

## Goal

Retire `fancy-regex` from lark-rs with **no runtime lookaround engine**. Every bounded
lookaround terminal is **eliminated at grammar-load time** — rewritten to an
equivalent lookaround-free regex that runs on the `regex` crate and rejoins the fast
combined-DFA scan. Anything that cannot be eliminated is **rejected at build time**
with a clear, actionable error.

This is "Option 1b" from the strategy memo. The Pike-VM engine explored in the closed
PR #110 is **not** part of this plan; it is shelved as the Option-H fallback, to be
revisited only if a real, irreducible-but-valid bounded lookaround grammar ever
surfaces (none exists in the two-corpus census; see the memo §10).

### Why (one paragraph)

A bounded assertion is regular, so for every grammar that exists it can be expressed
without lookaround and run on the one fast linear engine we already depend on. This
wins linearity (no ReDoS), average-case speed (rejoins the single combined-DFA pass
instead of an N-pass per-terminal side-probe), zero new parity surface, and — as a
bonus — makes the bundled `python`/`lark` grammars **standalone/WASM-bakeable** for
free (their terminals become pure `regex`), closing a standing follow-up. The cost is
a loud reject for the *empirically empty* set of novel, non-template bounded
lookaround. See the memo for the full argument and the impossibility triangle.

## Milestones

### E1 — Lock behavior (oracle matrix) ✅ *landed in this PR*

The engine-agnostic behavioral gate (`tests/test_lookaround.rs` +
`fixtures/oracles/lookaround/cases.json`, generated by
`tools/generate_oracles.py::generate_lookaround`). Records Python Lark's actual result
for each of the four boundary-assertion forms + the length-changing trailing lookahead
+ inline/global-flag cases, across the full **(parser × lexer) matrix** (lalr×{basic,
contextual}, earley×{basic,dynamic,dynamic_complete}, cyk×basic). It **passes on
today's `fancy-regex` lexer**, so it locks the semantics any rewrite must reproduce and
cannot silently diverge. Salvaged from PR #110 (it is independent of *how* the behavior
is produced).

### E2 — Rewrite the bundled terminals

> **E2a — landed (with one terminal proven irreducible).** `python.lark`'s
> `LONG_STRING` and the classic block-comment shape are rewritten lookaround-free and
> proven **byte-for-byte equivalent** to their originals by `tests/test_lookaround.rs`
> (per-terminal **match-length equivalence**: original on `fancy-regex` vs rewrite on
> `regex`, full agreement over a generated corpus) plus the stdlib oracle. `LONG_STRING`
> needs no priority change — its body has no boundary guard, so the lazy escaped form is
> a clean equivalent.
>
> `STRING` **resists elimination** and is left verbatim on lookaround. The mandatory
> equivalence proof is what surfaced this: `STRING`'s `(?!"")` opening guard is a
> *trailing-context* boundary — it makes `""""` a lex error while `"" ""` (two empty
> strings) is valid, a distinction that exists **only at lex time** (once `%ignore`
> drops the whitespace, both are `STRING STRING` to the parser, so no grammar-level or
> priority fix can recover it). Every lookaround-free rewrite therefore accepts `""""`,
> diverging from the oracle — *not acceptable*. This is exactly the "bundled terminal
> with no clean lookaround-free form … resists both regex rewrite and priority" risk
> below; per that risk it is **surfaced, not worked around**. The negative result (the
> standard rewrite is inequivalent) is pinned by
> `string_lookaround_free_rewrite_is_not_equivalent`. `STRING` stays on `fancy-regex`
> pending the E3/E4 decision on the irreducible set, which now contains a real member —
> see Risks. Still open in E2: `lark.OP`/`REGEXP`, `common.DEC_NUMBER` (the
> `(?![1-9])` / `(?![a-z])` boundary-as-failure suspects).

Replace the lookaround in the bundled grammars (`src/grammars/python.lark`,
`lark.lark`, `common.lark`, plus the `examples/` block-comment shape) with
lookaround-free equivalents, so they rejoin the combined scanner. **Each rewrite must
be proven equivalent**, not eyeballed — via the E1 oracle *and*, ideally, DFA
match-length equivalence against the original (build both terminals' anchored DFAs with
`regex-automata` and compare the matched-prefix function at every position; both are
regular, so this is decidable).

Candidate rewrites (all **to be verified**, not asserted):

| Terminal | Original (lookaround) | Candidate lookaround-free form | Notes |
|---|---|---|---|
| `STRING` (python) | `…("(?!"").*?(?<!\\)(\\\\)*?"\|'…')` | ~~`…("(?:[^"\\\n]\|\\.)*"\|'…')`~~ **rejected** | E2a result: candidate disproven — accepts `""""` (oracle rejects). The `(?!"")` is an irreducible trailing-context boundary; **stays on lookaround**. See the E2a note above |
| `LONG_STRING` (python) | `…""".*?(?<!\\)(\\\\)*?"""…` | `…"""(?:[^\\]\|\\.)*?"""…` (lazy, no newline exclusion) | ✅ E2a: landed, proven byte-for-byte equivalent (no priority change needed) |
| `REGEXP` (lark) | `\/(?!\/)(\\\/\|\\\\\|[^\/])*?\/[imslux]*` | `/(?:\\.\|[^/\\])+/[imslux]*` | drop `(?!\/)` by requiring a non-empty body |
| `OP` (lark) | `[+*]\|[?](?![a-z])` | fold into terminal **priority** vs the `?rule` modifier | likely *not* a regex rewrite — see below |
| `DEC_NUMBER` (common) | `…(?![1-9])` leading-zero guard | verify; may need priority or a negated-char construction | the lookahead turns a short match into a *failure* — the awkward case |
| `MULTILINE_COMMENT` (verilog, examples) | `\/\*(\*(?!\/)\|[^*])*\*\/` | `/\*(?:[^*]\|\*+[^*/])*\*+/` | the classic regular block-comment form |

**Known subtleties E2 must catch (why the equivalence proof is mandatory):**

- **Newlines.** Python's `.` (no DOTALL) excludes `\n`, but `[^"\\]` *includes* it. A
  naive `"(?:[^"\\]|\\.)*"` would let a short string span lines, diverging from
  Python. The newline must be excluded explicitly (`[^"\\\n]`). This is the canonical
  example of why eyeballing fails.
- **Boundary-as-failure (`OP`, `DEC_NUMBER`).** A trailing `(?![a-z])` / `(?![1-9])`
  that converts a *successful short match* into a *whole-terminal failure* often has no
  clean inline regex form. The right tool is usually **terminal priority** (let the
  longer/competing terminal win) or the `unless` keyword-retype machinery — i.e. a
  grammar-level fix, not a regex rewrite. E2 should expect a couple of these to move to
  priority rather than regex.
- **Lazy vs greedy ends.** The original uses lazy `.*?`; the rewrite is greedy but
  unambiguous (e.g. `[^"\\]` cannot eat the closing quote). Verify greedy and lazy
  coincide on every oracle case.

Update `CLAUDE.md`'s "Grammar standard library" and the `regex`-crate gotcha note: the
grammars stop being byte-identical to upstream here (the conscious trade from the memo,
axis 3), so document the divergence and the equivalence proof that backs it. (Today
only `common.lark`'s `ESCAPED_STRING` carries such an adaptation; E2 generalizes it.)

### E3 — Detect + reject hand-written lookaround

For *user* grammars (not the bundled ones), detect lookaround in a terminal and either
rewrite it (E5, optional) or **reject it loudly**. Reuse the lookaround front-end
parser from the closed PR #110 (`src/lookaround/mod.rs`) — its `Node` tree + assertion
extraction is exactly the detector/classifier a rewriter or rejecter needs; it is
engine-agnostic and does **not** depend on the Pike-VM (`matcher.rs`). Salvage it,
drop the `pub mod matcher;` line, and wire it as the classifier.

The reject must be a `GrammarError` that names the terminal, shows the offending
assertion, and suggests the fix (rewrite to lookaround-free form, use a rule, raise
terminal priority, or import the stdlib terminal). Add an oracle/test that a
representative unsupported pattern is rejected with that message.

### E4 — Remove `fancy-regex`, shelve the engine

Only after E2 lands (else the bundled grammars break). Remove the `fancy-regex`
dependency and the `AnyRegex::Fancy`/`Lowered` lexer routing; the lexer is then
`regex`-only. Confirm the bundled `python`/`lark` grammars now bake into the
standalone/C/WASM runtimes (the axis-6 bonus). Update `CLAUDE.md` (remove the
`fancy-regex` routing note; record that lookaround terminals are eliminated at load
time).

### E5 — *(optional, deferred)* template auto-rewriter

A small load-time pass that rewrites the common **hand-written** idiom classes the
census found — the escaped-string/block-comment idioms, negated-char lookahead
(`(?!set). → [^set]`), reserved-word exclusion (`(?!(KW|…)) →` `unless`/priority),
fixed-width lookbehind — so a user who hand-writes one of these (rather than importing)
gets it rewritten instead of rejected. Only worth doing if real reject-reports appear;
otherwise E3's loud reject is sufficient (the census shows the rejected set is empty).

## Verification strategy

- **E1 oracle matrix** gates behavior across all parser/lexer combos, on every change.
- **DFA match-length equivalence** (E2) upgrades the per-terminal rewrites from
  "tested on N cases" to "proven equivalent as matched-prefix functions" — available
  precisely because we stay in the regular world.
- **Oracle freshness gate** (CI) keeps `cases.json` in lockstep with Python Lark.

## Risks / open questions

- **A bundled terminal with no clean lookaround-free form.** `OP` and `DEC_NUMBER` are
  the suspects; mitigation is the grammar-level (priority/`unless`) route. If one truly
  resists both regex rewrite and priority, that is the first real signal to reconsider
  the Option-H fallback — surface it loudly rather than working around it.
- **Oracle/​lark version drift.** `cases.json` is generated against a pinned Python
  Lark; keep CI's lark version aligned (regenerated 2026-06-07 against lark 1.3.1).
- **Unbounded-width lookahead** (e.g. `strictdoc`'s `(?![ ]*X)`) is rejected for the
  hard linear guarantee — the one slice of Python compatibility deliberately given up
  (memo §6/§7).
